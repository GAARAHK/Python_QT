# 20路电机串口群控与数据采集系统 - 需求与实现说明文档

## 一、 系统概述
本系统主要用于通过双路 RS-485 总线实现对20路电机设备的集中控制、状态监控、工况编排、安全保护及数据追溯。系统核心流程为：手动定位文本框扫码接入 -> 绑定预设运行工况 -> 电机按设定自动循环运行 -> **按指定循环次数精准触发电流采集** -> 异常电流报警 -> 历史数据查询与导出。

## 二、 核心功能需求

1. **电机群控（COM_A）**：上位机通过单根 485 总线控制20个单机模块进行正转、反转、停止操作。每个模块分配独立设备 ID。
2. **电流采集（COM_B）**：上位机通过另一根 485 总线连接“20路电流模拟量读取模块”，用于读取各对应通道的电机运行电流。
3. **流程绑定**：每台电机对应一个实体二维码。操作人员通过点击输入框，手动使用扫码枪键入二维码后，为该电机分配对应的运行状态。
4. **工况配置与保存**：
   - 模式设定：可自由组合“正转/停止/反转”等动作，且每段动作的时长均可独立设置。
   - 循环设定：可设定目标循环次数。
   - 本地持久化：配置好的工况可保存在本地，软件下次启动或断电重启后自动加载历史设置。
5. **精准按需数据采集机制**（核心逻辑更新）：
   - 不采用高频无脑轮询。用户可设定“采集间隔”（如每隔1次、10次、100次循环采集一次）。
   - **触发时机**：当当前电机运行到设定的目标循环次数时，在“正转开始后延迟 2~3 秒”的时间点触发电流读取指令。
   - **目的**：避开电机启动瞬间的浪涌电流（防止误报警），同时精准同步当前电流值与“第几次循环”的映射关系。
6. **边界报警与停机**：在触发电流采集的瞬间，将取得的数据与设定的上下限安全范围对比。若超出范围，系统立即向该名录电机下发停止运行指令，并记录报警日志。
7. **数据记录与追溯导出**：
   - 使用 SQLite 轻量级数据库持久化记录。
   - 支持根据“二维码”对历史电流记录、运行状态等信息进行查询。
   - **数据导出**：系统需具备数据导出接口，支持将查询到的记录导出为 Excel 或 CSV 等通用报表格式。
8. **UI 与监控看板**：
   - 主界面设计以功能性与状态可视化为主，清晰展示各个设置界面。
   - 具备“20路电机一览大屏”，直观显示每一路的二维码信息、当前工况、当前运行循环数以及最新的电流状态/报警状态。

---
**需要注意的坑与风险评估**
硬件通信并发与时延冲突（重点）：
20个电机挂在同一根485控制线上，如果多个电机同时达到工况切换点（例如同时要求反转），串口总线不能并发只能排队，可能存在几十到几百毫秒的时延。应对手段：增加串口指令调度队列，保证指令有序发送，且增加超时重发机制。

## 三、 系统流程与控制设计思路

### 1. 通信与并发隔离
- 控制线程与采集线程分离，互不干扰。
- 由于采集策略更改为“按循环次数触发”，当同一时刻有多个电机同时到达设定循环（例如都跑到第100次），上位机会向串口压入一次“读取电流通道”指令，获取后分别分发给到达指定循序的电机作为当次记录。

### 2. 状态机与防浪涌采集逻辑
每路电机在上位机中维持一个虚拟状态机。以“隔100次采集，正反转模式”为例，其逻辑为：
- `步骤1`: 发送正转 -> 开始正转计时。
- `检查`: 当前是否为需要采集的周期（如 第100、200次...）？如果是，则触发“等待时间阀值（2~3秒）”的内部定时。
- `步骤2`: 达到 2~3 秒阀值，向采集模块索要当前电流并落盘，校验是否触发报警。若超标 -> 发送停止指令 -> 状态转迁为 `Error/Alarm` 终止运行。
- `步骤3`: 正转时间结束，发送停止 -> 停止计时。
- `步骤4`: 发送反转 -> 反转计时 -> ...完成单次循环。

### 3. 断电与异常恢复机制
系统不做运行状态的断点保存（即不记忆意外断电时电机跑到第几次）。设计逻辑如下：
- 系统启动时，**仅自动恢复“软件侧”的参数配置**（波特率、设备参数、预设的工况库档、报警上下限）。
- 各路电机的运行进度、在线状态需人工重新确认、扫码下发指令，从第 0 个循环重新开始安全运行。

## 四、 数据库分表设计 (SQLite)

鉴于单机长期运行数据量较大，推荐一套高效稳定的分表策略：

* **表1. `settings_config` (参数配置表)**：存放软件全局设置、串口配置参数等（覆盖断电恢复需求）。
* **表2. `work_mode_templates` (工况模板表)**：存放用户保存的各项自定义工况流（名称、各阶段时长、目标循环数、报警阈值、采集间隔）。
* **表3. `motor_run_history` (批次运行主表)**：每次扫码开始运行生成一条主记录 UUID，包含：条码、使用的模板、开始时间、最终结束状态（正常完成/报警停止/人工中止）。
* **表4. `current_logs_202x_xx` (电流详情日志流水表，建议按月建表)**：
  - 字段：`id`, `batch_uuid` (外键关联表3), `motor_id`, `qr_code`, `loop_count` (当前循环次数), `read_current` (实测电流值), `timestamp`。
* **表5. `alarm_logs` (报警故障表)**：专门用于独立展示报警追溯。包含产生报警的时刻、当时的限定阈值、实际超标电流和引发的具体动作。

## 五、 后续开发与部署确认事项

1. **权限架构**：单级权限，无用户角色区分（管理员/操作员），当前功能全员开放。
2. **硬件对接前置条件**：需硬件端尽快提供 485 控制单机模块通信协议以及 20路电流采集模块通讯协议（建议为主流 Modbus RTU 以缩短开发周期）。
3. **UI 侧重**：优先保障“二维码手动输入光标锁定防串位”、“20路列表无卡顿刷新”、“Excel 快速导出”几个核心卡点的用户体验。

注意事项
地址一致性：两个设备默认地址均为1，同时测试时需修改其中一个的地址（例如将电机控制器改为2）。

波特率一致：确保所有设备波特率、校验位、停止位设置相同。

CRC校验：表中CRC已计算正确，可直接使用。

指令格式：所有报文均为十六进制，发送时需按字节发送（高低字节顺序不变）。

如果需要修改其他参数或定制指令，可参考对应说明书中的寄存器定义。
